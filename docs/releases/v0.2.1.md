# v0.2.1 - preview feedback floor (patch)

**Tag:** v0.2.1 (pending) ┬Ę **Previous:** v0.2.0 ┬Ę **Date:** 2026-07-18

## Summary

Patch release. Fixes a silent false-pass where `preview_prototype` returned non-actionable feedback (empty, or "Õ«ēÕĖłÕż¦"-style unrelated anchors) yet still wrote `confirmed=true` and advanced to Fill. Discovered by dogfood 0015.

No new public API; the preview adapter's confirm contract tightens (a confirm now requires substantive feedback). Pre-ADR confirm records now fail G5 by design (see Breaking change).

## Why

Dogfood 0015 (`dogfood/2026-07-18-0015.md`, G1, high) showed a preview confirm with garbage feedback sailing through to Fill - a silent false-pass violating the CONTEXT principle "Evidence exists only to satisfy a declared criterion." The pill's primary "ńĪ«Ķ«żķĆÜĶ┐ć" button was `type="submit"` and submitted an empty form; G5 only checked that a confirm record existed, not that its feedback was substantive.

## What changed

**Two-layer floor (ADR-0008):**

1. **Adapter structural floor** (`packages/design-playbook-preview/server.py`) - `_check_feedback_floor` runs before `_write_confirm`. Passes when: non-empty feedback OR Ōēź1 anchor present (trigger), AND every present anchor carries a non-empty `selector` AND a non-empty `comment` (completeness, independent of feedback). On failure: writes `confirmed=false` + `floor_failure` reason; the run recirculates to revise instead of advancing to Fill.
2. **Frontend defense-in-depth** (in-page JS, same semantics) - the pill's confirm/revise submit now `preventDefault`s a non-substantive submit, opens the drawer, and surfaces a hint. Same floor for UX; the adapter floor remains the authoritative backstop.
3. **G5 gate** (`scripts/validate_run.py` `check_preview`) - confirmed records now require `floor_pass: true`, not just `confirmed=true`. Rejects with `failed feedback floor`. No new gate (issue 04 still bans G7); enforced inside the existing G5 path.
4. **Evaluator semantic judgment** (`ui-evaluator` rubric) - reads `preview/log.md` + confirm json as a supporting finding, `source = preview* seam`, catching what the structural floor cannot (e.g. "Õ«ēÕĖłÕż¦" is non-empty but semantically unrelated).

**Floor semantics are unified across all three sites** (adapter Python, frontend JS, ADR-0008) - a two-axis code review caught them diverging (`any` vs `every comment-only` vs `every selector AND comment`); all three now agree.

**Also:**
- Q3.2 (manifest `observed` field) - confirmed already enforced by G6 (`evidence/` prefix); no code work. The dogfood gap was a wording error, not a product defect.
- Q3.4 (craft hover affordance) - `craft-guard` Done-when now requires every L4 interactive region to declare motion/hover purpose (no silent missing-hover PASS).
- **i18n** (`packages/design-playbook-preview/i18n.py`) - the preview adapter confirm UI (pill, drawer, hint, buttons, done page) is now locale-aware: `DPB_PREVIEW_LANG` (or `LANG`), default zh-CN, with `en` provided. `CONFIRM_LABELS` / `REVISE_LABELS` are cross-locale unions so the backend still classifies confirm/revise regardless of UI language.

## Breaking change (by design)

Pre-ADR confirm records (written before this release) carry no `floor_pass` field and now **fail G5** with `failed feedback floor: no floor_pass=true`. This is intentional: a confirm that cannot prove its feedback passed the floor is not authoritative evidence. To re-validate a pre-ADR run, re-run its preview step rather than patching the old record. v0.2.0 shipped days prior; the install base of in-flight runs is negligible.
